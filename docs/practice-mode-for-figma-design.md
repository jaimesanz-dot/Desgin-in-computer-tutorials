## Problem Statement

People learn complex software badly because most guidance lives outside the moment of action. They watch a video, read a tutorial, switch back to the app, forget the step, and repeat. The idea here is a guidance layer inside the computer that shows what to do next and helps the learner practice while doing the real task.

For the hackathon demo, the first audience is people learning serious software or creative tools. The first app is Figma. The first workflow is building a simple reusable UI component.

## What Makes This Cool

The "whoa" moment is that the guide is not just a tour. It is a practice coach. It points to the next action, watches what the learner creates, and corrects them when they are off track.

The sharp framing is:

> Practice mode for software.

Existing product-tour tools mostly help companies onboard users or drive feature adoption. This should feel different: a coach that helps a person build skill in a creative tool.

## Constraints

- Hackathon/demo mode, so the experience must be showable fast.
- One app first, not universal coverage.
- One workflow first, not a library of lessons.
- The demo should feel real, so it must validate the learner's actual Figma document state.
- The universal "system of guidance inside computers" vision should appear in the story, but not be required for the first build.

## Premises

1. The hackathon demo should use one app and one workflow, not "any app" yet.
2. The first app should be Figma, with a simple component-building lesson.
3. The magic should be learn plus practice at the same time: highlight next action, observe user action, correct mistakes.
4. The demo should fake or constrain universal intelligence where needed, because the real wow is the coaching experience, not perfect automation across every app.
5. The end state should be something visual the user made, so the audience sees progress, not just instructions.

## Landscape Notes

- Appcues and similar tools focus on in-app messaging, product onboarding, feature adoption, and behavior-based customer engagement.
- WalkMe and digital adoption platforms focus on enterprise workflows, training, analytics, automation, and reducing software friction across business applications.
- Research systems like HelpViz show that contextual visual tutorials can outperform text-only instructions.
- Newer research like GuideWeb shows automatic guide generation is still difficult, especially selecting the right UI target and generating correct guidance.

The opportunity is not "make another product-tour builder." The opportunity is to make a practice coach for creative software.

## Approaches Considered

### Approach A: Browser Overlay Coach

Summary: Build a floating guide over Figma in the browser. It highlights areas, shows the next instruction, and advances through a scripted lesson.

Effort: Small
Risk: Medium

Pros:
- Fastest path to a polished visual demo.
- Makes the "guidance inside your screen" idea obvious immediately.
- Easy to explain to a hackathon audience.

Cons:
- Correction may be fake or shallow.
- If Figma's UI changes or the window moves, the illusion can break.
- It proves the overlay more than it proves learning.

Reuses:
- Browser extension patterns.
- Fixed hotspots.
- Scripted tutorial flow.

### Approach B: Figma Plugin Practice Coach

Summary: Build a Figma plugin that teaches one workflow, like creating a button component, and checks the actual Figma document state after each step.

Effort: Medium
Risk: Medium

Pros:
- The coaching is real because the plugin can inspect layers, frames, colors, sizing, names, and component state.
- The learner ends with a real finished component.
- It proves the core loop: instruct, observe, correct, advance.

Cons:
- It proves the idea inside Figma first, not across every app.
- The plugin UI has less freedom than a full desktop overlay.
- Validation rules need to be carefully scoped so they do not feel brittle.

Reuses:
- Figma plugin API.
- Step checklist.
- Document validation rules.

### Approach C: Ghost Mouse Lesson Recorder

Summary: Record an expert doing a workflow, then replay it as a ghost cursor with highlights while the learner follows along. Add simple "you are off track" detection.

Effort: Large
Risk: High

Pros:
- Feels closest to the big vision of learning any software from expert demonstrations.
- Very memorable as a demo.
- Could later turn expert workflows into shareable lessons.

Cons:
- Harder to make reliable in a hackathon.
- Validation may depend on screenshots or brittle action matching.
- It may become a playback demo rather than a true practice coach.

Reuses:
- Screen recording.
- Cursor replay.
- OCR or computer vision.
- Tutorial playback.

## Recommended Approach

Choose Approach B: Figma Plugin Practice Coach.

This best proves the core idea: learning and practicing at the same time. It narrows the first demo enough to build, but keeps the big vision intact. The demo can truthfully say: "The system is checking what you made, not just playing a tour."

## First Lesson

Lesson title: Build a reusable button component.

Target outcome: The learner creates a button component with a frame, text label, fill color, padding, rounded corners, and component status.

Lesson architecture:

- Define the lesson as a typed array of step objects.
- Each step includes `id`, `title`, `instruction`, `hint`, `validatorId`, `successMessage`, and `correctionMessages`.
- Keep validator implementations as named functions mapped by `validatorId`.
- Every validator returns the same result shape: `{ status, message, details, nextAction }`.
- Valid `status` values are `pass`, `fail`, and `near-miss`.
- The panel renders validator results generically instead of branching on each lesson step.
- Do not build a full lesson authoring format for v1.
- Do not hard-code the whole lesson directly inside plugin control flow.

Step flow:

1. Create a new frame for the button.
2. Set the frame size, for example 160 by 48.
3. Add text: "Get started."
4. Center the text inside the frame.
5. Apply a fill color.
6. Add corner radius.
7. Rename the layer to "Button / Primary."
8. Convert it into a component.
9. Show completion state with a small celebration and summary.

Correction examples:

- If the learner creates a rectangle instead of a frame: "Good start, but use a frame so we can turn this into a flexible component."
- If the text is missing: "Add a label so the button communicates an action."
- If the size is far off: "This is too small for a primary button. Aim for about 160 by 48."
- If it is not a component yet: "Final step: make this reusable by turning it into a component."

## Interaction Model

The plugin should have a compact side panel:

- Current step title.
- One short instruction.
- "Check my work" button.
- Progress indicator.
- Correction message when the document does not match.
- "Show hint" option for learners who are stuck.

Validation model:

- The plugin validates only when the learner clicks "Check my work."
- The plugin does not pretend to run as a background watcher.
- This matches Figma's plugin model and keeps the hackathon demo reliable.
- Validation is selection-first: inspect the selected node/subtree before scanning elsewhere.
- If no useful node is selected, fall back to a small named search for `Button / Primary`.
- Avoid whole-page validation scans during normal checks.
- Later polish may add selection-aware hints while the plugin panel is open, but automatic background coaching is out of scope for v1.

The canvas should remain the main stage. The learner should feel like they are using Figma, not taking a quiz inside a separate app.

## Success Criteria

The demo is done when:

- A first-time viewer understands the product in under 20 seconds.
- A learner can complete the button lesson in under 3 minutes.
- The plugin detects at least 5 real document conditions, not just button clicks.
- The plugin gives at least 3 useful corrections.
- The learner ends with a visible finished Figma component.
- The pitch clearly explains the bigger vision: this could become practice mode for many tools.

## Test Strategy

Testing is in scope for the first build.

- Use Vitest unit tests for the lesson state machine and validator functions.
- Build an automated plugin/integration harness for the core lesson flow.
- Use fake Figma node objects for fast validator tests.
- Use at least one automated end-to-end style run that simulates: launch lesson, fail a step, receive correction, fix the document state, pass the step, and reach completion.
- Keep one manual Figma smoke test for final demo confidence.
- Treat validator false positives as release blockers.

## Distribution Plan

For a hackathon, distribution can be simple:

- Local Figma plugin loaded in development mode.
- Demo video or live walkthrough showing the lesson.
- Public GitHub repo if the team wants judges or friends to inspect it.

If continued after the hackathon:

- Publish to the Figma Community as a plugin.
- Add a small lesson library.
- Create a lesson authoring format so experts can define workflows.
- Later explore browser extension or desktop overlay versions for tools outside Figma.

## Open Questions

- Should the plugin teach one polished lesson or two shorter lessons?
- Should the correction feel strict, like a test, or friendly, like a coach?
- Should the guide use a voice/personality, or stay minimal and tool-like?
- How much of the future "universal layer" should appear in the demo UI versus the pitch?

Accepted decisions:

- Validation trigger: explicit "Check my work."
- Lesson shape: typed step array with named validators.
- Validator result: shared `{ status, message, details, nextAction }` contract.
- Validation scope: selected node/subtree first, `Button / Primary` fallback only.
- Test scope: Vitest unit tests plus an automated plugin/integration harness.

### Failure Modes

| Codepath | Failure mode | Test required | Error handling | User-visible result |
|----------|--------------|---------------|----------------|---------------------|
| Plugin launch | Figma API unavailable or wrong editor mode | yes | show setup/error message | clear error |
| Lesson state | step id missing or validator id unknown | yes | fail closed with developer-facing message | clear error |
| Validator lookup | selected node is wrong type | yes | return fail with correction | clear correction |
| Size validator | values nearly match but not exact | yes | return near-miss with tolerance details | clear correction |
| Text validator | text node missing or nested unexpectedly | yes | search selected subtree before failing | clear correction |
| Component validator | object is visually correct but not componentized | yes | fail with next action | clear correction |
| Completion state | learner edits an earlier passed step | yes | revalidate current requirement before completion | clear correction |

No silent critical gaps are allowed.
